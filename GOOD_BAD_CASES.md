### 2026-04-25 — Таймер трекера: разбор дат с API
- Инициатор: User / AI
- Контекст: `00:00` при открытии, обратный отсчёт при кривом якоре, лишние `Activity.update` и шумы в таймере; на симуляторе нестабилен `TimelineView` в ряде iOS-версий; опорный рабочий iOS-коммит `69750d0` зафиксировали как эталон тика.
- Предложение: единый robust-парсинг ISO от backend; никогда не подменять `sleepStart`/`lastWake` на `Date()` при ошибке; опционал для `referenceDate` в UI.
- Варианты:
  - Вариант A: оставлять `ISO8601DateFormatter()` в одной строке.
  - Вариант B: многошаговый `TrackerAPIDate.parse` + сохранение прежнего якоря при сбоях.
- Выбор пользователя: B
- Ожидаемый эффект: дашборд и LA показывают тот же интервал, что и сервер; нет фантомного нуля.
- Фактический результат: сброс «нуля» снят; в долгую — наблюдение. Локализация LA/прочие визуальные правки — не смешивать с фиксом тика.
- Кейс: GOOD (B)
- Вывод: разбор дат в одном стиле с backend — обязателен; `Date()` как fallback для якоря = источник багов. Разделать «частые обновления LA» и «стабильный тик» отдельной задачей, после стабилизации якоря.

- Варианты:

- Выбор пользователя: B

- Фактический результат: контракт восстановлен, backend tests (`test_budget`, `test_tracker`) проходят.
- Кейс: GOOD
- Вывод: при работе через feature-ветки обязательно проверять, какие доменные изменения уже попали в `main`.

### 2026-04-23 — Усиление логина для семейного приложения
- Инициатор: User
- Контекст: концепция "общие данные семьи", но недопустим открытый вход по любому имени.
- Предложение: перейти на парольный вход и закрытую регистрацию.
- Варианты:
  - Вариант A: оставить логин только по имени.
  - Вариант B: `name + password`, создание пользователей только админом, forced reset первого пароля.
- Выбор пользователя: B
- Ожидаемый эффект: сохранить shared-модель данных и убрать несанкционированный доступ.
- Фактический результат: backend+iOS переведены на парольный вход, добавлен admin bootstrap и forced reset flow.
- Кейс: GOOD
- Вывод: для семейного shared-доступа лучше усиливать аутентификацию, а не дробить данные по пользователям.

- Инициатор: AI
- Контекст: при внедрении auth-hardening был создан новый `.env` в корне проекта.
- Предложение: быстро создать env с admin-параметрами без уточнения текущего расположения.
- Варианты:
  - Вариант A: сначала проверить существующий env и уточнить у пользователя, куда писать.
  - Вариант B: создать новый env автоматически.
- Выбор пользователя: A (задним числом как правильное поведение)
- Ожидаемый эффект: единая точка конфигурации и отсутствие путаницы.
- Фактический результат: сначала создан лишний корневой `.env`, потом исправлено переносом в `backend/.env` и удалением дубликата.
- Кейс: BAD
- Вывод: перед созданием/переносом конфигов всегда сначала проверять существующую структуру и спрашивать подтверждение по целевому месту.

### 2026-04-24 — Миграция схемы бюджета в коде (без «ручного voodoo» на каждом сервере)
- Инициатор: User + AI
- Контекст: на production осталась старая таблица `categories` без `group_id`; `import_history` и ORM падают, потому что `create_all` не меняет существующие таблицы.
- Предложение: детектить отсутствие `group_id` на старте и пересоздать бюджетные таблицы, затем полагаться на `git pull` + рестарт.
- Варианты:
  - Вариант A: каждый раз помнить сырые `DROP/ALTER` в консоли на сервере.
  - Вариант B: `ensure_budget_schema()` в `main.py` + задокументировать сценарий.
- Выбор пользователя: B
- Ожидаемый эффект: старые SQLite-файлы сами доводятся до контракта, меньше человеческих ошибок.
- Фактический результат: после pull и `restart` cloud выровнен; реимпорт истории снова возможен; локал и cloud на одной логике.
- Кейс: GOOD
- Вывод: лёгкие, измеримые миграции рядом с `create_all` — разумно для single-file SQLite, пока нет нормального миграционного движка.
