# AI Insights Flow (PVLFamily)

## Текущий триггер вызова LLM

1. Пользователь открывает экран аналитики:
   - бюджет: `BudgetAnalyticsHubView`
   - трекер: `TrackerAnalyticsHubView`
2. Экран загружает агрегаты с backend (как раньше).
3. После успешной загрузки агрегатов экран формирует `safe_payload` и вызывает:
   - `POST /insights/budget`
   - `POST /insights/tracker`
4. Если LLM-ответ успешный, в карточках "Сводка" текст заменяется на LLM-версию.
5. Если LLM недоступна, остаётся текущий rule-based текст на клиенте.

## Что отправляется в LLM

В backend endpoint `/insights/*` передаётся только агрегированный payload:

- тип отчёта и период;
- числовые метрики (`metrics`);
- флаги трендов (`trend_flags`);
- аномалии в агрегированном виде (`anomalies`);
- без сырых транзакций, комментариев и персональных строк.

## Что выводится пользователю

На экранах аналитики выводятся:

- `summary_today` (краткий вывод на сегодня),
- `summary_month` (краткий вывод за период),
- подпись `LLM: <provider>`.

Если LLM не ответила — остаются локальные эвристики.

## Что нужно настроить

1. Получить API-ключи нужного провайдера.
2. Заполнить `backend/.env` по образцу `backend/.env.example`.
3. Выбрать провайдер:
   - `LLM_PROVIDER=openai|deepseek|qwen|anthropic|gigachat|auto`
4. Перезапустить backend.

## Локаль vs облако

- **Локально (dev):** backend на Mac, ключи в локальном `backend/.env`.
- **Облако (prod):** ключи и `LLM_PROVIDER` задаются в `.env` на сервере `pvlfamily`, затем restart сервиса.

## Рекомендуемый rollout

1. Проверка локально с `LLM_PROVIDER=deepseek` или `openai`.
2. Сверка формата ответов и latency.
3. Отдельный деплой на cloud после проверки.
4. В cloud сначала ограниченный запуск (только бюджет или только трекер), затем полный.
