## Сессия: Экспорт данных и работа над ошибками компиляции
**Дата:** 2026-04-17
**Цель:** Реализовать выгрузку данных в Excel через интерфейс iOS приложения.

### Хронология действий:
1. **Бэкенд (Успех):**
   - Реализованы эндпоинты `/budget/export/excel` и `/tracker/export/excel`.
   - Добавлена фильтрация по датам, разделение категорий на колонки, стилизация таблиц.
   - Установлена зависимость `openpyxl==3.1.5`.
   - *Инцидент:* Первая версия сломала сборку из-за несовместимости аргументов `TableStyleInfo`. Исправлено откатом стиля.

2. **iOS Интеграция (Провал -> Успех):**
   - **Попытка 1:** Прямое добавление кнопок в `ProfileView.swift`.
     - *Ошибка:* Компилятор Swift выдал `Generic parameter 'Content' could not be inferred` и `unable to type-check`. Причина: перегрузка типа в сложном `body`.
   - **Решение:** Откат к рабочей версии. Вынос логики экспорта в отдельный экран `ExportDataView.swift`.
   - **Результат:** Создан отдельный файл с выбором типа данных и периода. Навигация через `NavigationDestination`.
   - *Мелкая правка:* Иконка `doc.badge.arrow.down` заменена на универсальную `square.and.arrow.down` (iOS 17+ проблема).

3. **Деплой на сервер (Инцидент):**
   - Сервис упал с 502 ошибкой после обновления `requirements.txt`.
   - *Причина:* Скрипт деплоя не обновил зависимости в venv автоматически.
   - *Решение:* Ручная установка пакетов на сервере.

### Итоги сессии:
- ✅ Экспорт Бюджета и Сна работает.
- ✅ Интерфейс вынесен в отдельный экран `ExportDataView`.
- ❌ Неудачный опыт с усложнением `ProfileView`.

### Уроки (Lessons Learned):
1. **Правило "Разделяй и властвуй":** Если компилятор тупит на типизации в большом `body` — немедленно выноси функциональность в отдельный View.
2. **Проверка иконок:** Использовать только базовые SF Symbols или проверять версию iOS.
3. **Деплой зависимостей:** Всегда проверять логи сервиса (`systemctl status`) после изменения `requirements.txt`. Автоматика может молча пропустить установку.
4. **Сложные алерты:** Избегать `.constant(...)` в сложных иерархиях, использовать явные `@State`.

   - После обновления `requirements.txt` сервис на cloud.ru упал с ошибкой 502.
   - *Причина:* Скрипт деплоя не обновил зависимости в виртуальном окружении сервера автоматически.
   - *Решение:* Ручная установка пакетов через `pip install -r requirements.txt` внутри venv на сервере.

### Уроки (Lessons Learned):
1. **Правило "Разделяй и властвуй":** Если `View` становится сложным или компилятор начинает "тупить" на типизации — немедленно выноси новую функциональность в отдельный файл/экран. Не пытайся впихнуть всё в один `body`.
2. **Проверка иконок:** Всегда проверять доступность SF Symbols для целевой версии iOS. Если сомневаешься — используй классику (`square.and.arrow.down`).
3. **Деплой зависимостей:** Автоматические скрипты могут молча пропускать установку новых пакетов. После изменения `requirements.txt` всегда проверяй логи сервиса на сервере (`systemctl status`).
4. **Сложные алерты:** Избегать использования `.constant(...)` в `.alert` внутри сложных иерархий, если это вызывает ошибки вывода типов. Лучше использовать явные `@State` булевы флаги.
